How to Get sel4test Running on NVIDIA Jetson Orin Nano 8GB

A complete step-by-step guide for porting the seL4 microkernel to the NVIDIA Jetson Orin Nano 8GB Developer Kit. Covers hardware setup, serial console wiring, building a custom BL31 firmware to fix T234 RAS errors, cross-compiling sel4test, and booting from UEFI. All 141 compiled tests pass deterministically.

1. Overview

This project ports the seL4 microkernel to the NVIDIA Jetson Orin Nano 8GB Developer Kit, built around the Tegra T234 SoC (6-core Cortex-A78AE, ARMv8.2-A). seL4 boots bare-metal from UEFI as an EFI application, running the complete sel4test test suite.

What works: 141 out of 141 compiled tests pass. 42 tests are disabled at compile time (16 MCS-only, 7 SMP-only, 1 multi-domain, 1 permanently disabled, 1 VCPU/VM, and others not applicable to this non-MCS, non-SMP, single-domain configuration).

What you will end up with: A Jetson Orin Nano that boots seL4 from UEFI (via HTTP boot or SD card), runs the full sel4test suite, and prints results over a serial console.

The port required changes across 5 seL4 repositories plus a custom build of NVIDIA's BL31 (ARM Trusted Firmware) to handle a T234-specific RAS issue. All code is published on GitHub.

3. Serial Console: UARTC on J14 Button Header

The serial console uses UARTC on the J14 button header — the small header near the three buttons on the carrier board. It is not the 40-pin GPIO header.

Important: UARTC is in the always-on (AON) power domain. Unlike UARTA on the 40-pin header, UARTC's clock does not die when BPMP power-gates the peripheral. This is why UARTC was chosen.
J14 PinSignalWire ColorConnect to Adapter
Pin 3RXDGreenAdapter TX
Pin 4TXDWhiteAdapter RX
Pin 7GNDBlackAdapter GND
 
J14 Button Header (looking at the board, buttons nearest to you):

    Pin 1   Pin 2
    Pin 3   Pin 4    <- RXD (3), TXD (4)
    Pin 5   Pin 6
    Pin 7   Pin 8    <- GND (7)
    Pin 9   Pin 10
    Pin 11  Pin 12

    DO NOT connect VCC/3.3V from the adapter to the Jetson.

Serial parameters: 115200 baud, 8N1. Connect with:

picocom -b 115200 /dev/ttyUSB0

(Your device may be /dev/ttyUSB0, /dev/ttyUSB1, or /dev/ttyACM0. Use dmesg | tail after plugging in the adapter.)

4. Installing JetPack Linux

Before flashing custom firmware, the Jetson must have stock NVIDIA JetPack/L4T Linux installed. This ensures the QSPI flash has the correct firmware chain.

4.1 Download BSP

Download from developer.nvidia.com/embedded/jetson-linux (R36.4.4 or later):

  • Jetson_Linux_R36.4.4_aarch64.tbz2 (BSP, ~712 MB)
  • Tegra_Linux_Sample-Root-Filesystem_R36.4.4_aarch64.tbz2 (rootfs, ~1.8 GB)

4.2 Extract and Prepare

mkdir -p ~/jetson/linux && cd ~/jetson/linux

# Extract BSP
tar xjf Jetson_Linux_R36.4.4_aarch64.tbz2

# Extract root filesystem
sudo tar xjf Tegra_Linux_Sample-Root-Filesystem_R36.4.4_aarch64.tbz2 \
    -C Linux_for_Tegra/rootfs/

# Apply NVIDIA binaries
cd Linux_for_Tegra
sudo ./apply_binaries.sh

# Create default user
sudo ./tools/l4t_create_default_user.sh -u nvidia -p nvidia -a

4.3 Flash

cd ~/jetson/linux/Linux_for_Tegra
sudo ./flash.sh jetson-orin-nano-devkit mmcblk0p1

This flashes both the QSPI firmware and writes the rootfs to the SD card.

5. Custom BL31 Firmware (Required)

5.1 Why This Is Necessary

The T234 firmware chain (MB1, MB2, BL31, UEFI) leaves dirty cache lines in the System-Level Cache (SLC / L4) at protected physical addresses (notably paddr 0x0 and 0xda0). These are firmware-owned addresses that seL4 never uses.

Under cache pressure, the SLC evicts these dirty lines. The writeback hits the IOB (I/O Bridge), which generates a RAS (Reliability, Availability, Serviceability) error. This becomes an SError exception routed to EL3.

Stock NVIDIA BL31 treats all uncorrectable RAS errors as fatal: it powers off the CPU core. This is appropriate for automotive ASIL-D safety but fatal for seL4 — it kills the only running core and the system hangs.

Custom BL31 logs the error (it has already been cleared from the hardware error registers) and resumes execution. The error is harmless because the addresses involved are firmware-owned and never read by seL4.

Error path: SLC dirty line eviction → writeback → SCF → IOB → CBB bus error → IOB generates RAS → SError routed to EL3 → custom BL31 logs + clears + resumes.

5.2 Clone ATF Source

cd ~/jetson
git clone -b jetson_36.4.3 \
    https://nv-tegra.nvidia.com/tegra/optee-src/atf.git
cd atf/arm-trusted-firmware

5.3 Modify the RAS Handler

Edit plat/nvidia/tegra/soc/t234/plat_ras.c. Find the tegra234_ras_handler() function. At the bottom, there is a block that checks system_is_not_recoverable and powers off the core.

Stock code (remove this):

if (system_is_not_recoverable) {
    ERROR("Powering off core\n");
    (void)mce_command_handler(
        (uint64_t)MCE_CMD_ENTER_CSTATE,
        (uint64_t)CSTATE_CCPLEX_C7, ...);
    while (true) {
        wfi();
    }
}

Replace with:

/* For uncorrectable errors: log but do NOT power off the core.
 *
 * Stock NVIDIA BL31 powers off the core here, which is appropriate
 * for automotive safety (ASIL-D) but fatal for our seL4 use case.
 *
 * T234 firmware leaves dirty cache lines in the SLC (L4) at
 * protected addresses (paddr 0x0, 0xda0). When cache pressure
 * evicts them, writebacks trigger IOB RAS errors. These are
 * harmless -- the addresses are firmware-owned and we never
 * read them. The error has been logged and cleared above. */
if (system_is_not_recoverable) {
    ERROR("RAS: uncorrectable error handled, "
          "NOT powering off core (custom BL31)\n");
}
return 0;

5.4 Add Startup Banner (Optional)

Edit plat/nvidia/tegra/common/tegra_bl31_setup.c. After the call to plat_enable_console(), add: 

NOTICE("BL31: custom build (RAS no-kill)\n");

5.5 Build BL31

sudo apt install gcc-aarch64-linux-gnu

cd ~/jetson/atf/arm-trusted-firmware

make BUILD_BASE=./build-t234 \
     CROSS_COMPILE=aarch64-linux-gnu- \
     TARGET_SOC=t234 \
     DEBUG=0 \
     LOG_LEVEL=20 \
     PLAT=tegra \
     SPD=opteed \
     BRANCH_PROTECTION=3 \
     ARM_ARCH_MINOR=3 \
     LDFLAGS="--no-warn-rwx-segments" \
     -j$(nproc) \
     bl31

Output: build-t234/tegra/t234/release/bl31.bin (~95 KB).

5.6 Generate TOS Image

The QSPI "secure-os" partition contains a TOS (Trusted OS) image that bundles BL31, OP-TEE, and a DTB. Rebuild using NVIDIA's gen_tos_part_img.py: 

# Back up the stock image first
cp ~/jetson/linux/Linux_for_Tegra/bootloader/tos_t234.img \
   ~/jetson/linux/Linux_for_Tegra/bootloader/tos_t234.img.stock

# Rebuild with custom BL31 + stock OP-TEE + DTB
python3 ~/jetson/linux/Linux_for_Tegra/nv_tegra/tos-scripts/gen_tos_part_img.py \
    --monitor build-t234/tegra/t234/release/bl31.bin \
    --os <path-to-stock-optee.bin> \
    --dtb <path-to-stock-tos.dtb> \
    --tostype optee \
    ~/jetson/atf/tos-custom_t234.img

5.7 Flash to QSPI

Enter USB Recovery Mode

  1. Power off the Jetson.
  2. Hold the Force Recovery button (middle button on the carrier board).
  3. Power on while holding Force Recovery.
  4. Release after 2 seconds.
  5. Verify: lsusb | grep -i nvidia — should show "NVIDIA Corp. APX".
SD card must be inserted during flashing — MB2 initializes all storage including the SD card. Without it, flash hangs.

Flash Both A and B Slots

NVIDIA uses A/B redundant boot. Both slots must be updated or the device may boot from the unmodified slot. 

# Install custom TOS image
cp ~/jetson/atf/tos-custom_t234.img \
   ~/jetson/linux/Linux_for_Tegra/bootloader/tos_t234.img

cd ~/jetson/linux/Linux_for_Tegra

# Flash Slot A
sudo ./flash.sh -k A_secure-os jetson-orin-nano-devkit mmcblk0p1

# Re-enter recovery mode, then flash Slot B
sudo ./flash.sh -k B_secure-os jetson-orin-nano-devkit mmcblk0p1

Verify

Power cycle. On the serial console, look for:

NOTICE: BL31: custom build (RAS no-kill)

6.3 Build

cd ~/jetson/potanin-git
mkdir build-orin && cd build-orin

../init-build.sh \
    -DPLATFORM=orin-nano \
    -DAARCH64=TRUE \
    -DARM_HYP=ON \
    -DSIMULATION=FALSE \
    -DRELEASE=FALSE

ninja

Output: build-orin/images/sel4test-driver-image-arm-orin-nano — a PE/COFF EFI executable (~6 MB).

Clean rebuild required if you change the device tree overlay: rm -rf build-orin and re-run init-build.sh + ninja.

7. Booting seL4

Option A: HTTP Boot (Recommended)

HTTP boot lets you rebuild on the host and immediately reboot the Jetson without touching the SD card. 

# Serve the EFI binary from the host machine
mkdir -p /srv/tftp
cp build-orin/images/sel4test-driver-image-arm-orin-nano /srv/tftp/sel4.efi
cd /srv/tftp && python3 -m http.server 8080

One-time UEFI setup: Connect HDMI + keyboard. Press ESC during boot to enter UEFI Setup. Add an HTTP boot entry pointing to http://<host-ip>:8080/sel4.efi. Set it as the first boot option.

Option B: SD Card Boot

# Format SD card as FAT32, then:
sudo mount /dev/sdX1 /mnt/sd
sudo mkdir -p /mnt/sd/EFI/BOOT
sudo cp build-orin/images/sel4test-driver-image-arm-orin-nano \
        /mnt/sd/EFI/BOOT/BOOTAA64.EFI
sudo umount /mnt/sd

9. Technical Details

9.1 Boot Chain

BROM (on-chip ROM, EL3)
  → MB1 (first-stage bootloader, NVIDIA proprietary, EL3)
    → MB2 (second-stage bootloader, EL3)
      → BL31 / ATF (ARM Trusted Firmware, EL3 -- stays resident)
        → UEFI (NVIDIA EDK2-based, EL2)
          → elfloader (seL4 EFI application, EL2)
            → seL4 kernel (EL2) + root task (EL1)

UEFI runs at EL2, not EL1. seL4 must be built with ARM_HYP=ON. BL31 stays resident at EL3 handling SMC calls (PSCI, RAS) for the system's lifetime.

9.2 CPU

  • 6 Cortex-A78AE cores (ARMv8.2-A), silicon revision r0p1
  • 2 clusters: Cluster 0 (cores 0-3), Cluster 1 (cores 4-5)
  • seL4 runs single-core (core 0 only, non-SMP configuration)

9.3 Memory Map

RegionStartEndSize
Region 10x800000000xbe000000992 MB
Gap (CO:43)0xbe0000000xc200000064 MB
Region 20xc20000000x100000000992 MB

The gap at 0xbe000000 is firmware carve-out CO:43 (allocated by MB1, hardware-protected by the SNOC bus fabric). Any CPU access triggers an uncorrectable RAS error. Physical addresses 0x0–0x200000 are also reserved to prevent device untypeds covering firmware-protected addresses.

9.4 GIC and Timer

GICv3 (GIC-600AE): GICD at 0x0f400000, GICR at 0x0f440000. ARM Generic Timer at 31.25 MHz, PPIs: Secure=13, NS=14, Virtual=11, Hyp=10.

9.5 UART

UARTC at 0x0c280000, 8250-compatible, 32-bit register spacing. Clock ID 157, enabled once by elfloader via BPMP IPC. AON power domain — clock stays alive without keepalive.

9.6 Cache Maintenance

The T234 has a 4 MB System-Level Cache (SLC / L4). dc cisw (clean/invalidate by set/way) only reaches L1/L2/L3, not the SLC. The elfloader uses dc cvac (clean by VA to Point of Coherence). For PTE writes, the kernel uses dc civac + dsb (clean AND invalidate + full system barrier).

9.7 Key Build Configuration

PLATFORM=orin-nano
AARCH64=TRUE
ARM_HYP=ON                          # UEFI boots to EL2
SIMULATION=FALSE
RELEASE=FALSE                       # Enables debug output
KernelPlatformPASizeBitsOverride=40  # 40-bit PA (not default 44)

10. Troubleshooting

"Powering off core" — system hangs

Stock NVIDIA BL31 is running. Flash the custom BL31 (Section 5).

No serial output

  • Check wiring: J14 Pin 3 = RXD (adapter TX), Pin 4 = TXD (adapter RX), Pin 7 = GND.
  • Try swapping TX and RX.
  • Verify 115200 baud and 3.3V adapter (not 5V).
  • Ensure you're using J14, not the 40-pin GPIO header.

USB flash hangs

  • Try a different USB port / bus on the host machine.
  • Ensure SD card is inserted in the Jetson.
  • Verify recovery mode: lsusb | grep -i nvidia.

UEFI doesn't HTTP boot

  • Verify HTTP server: curl http://<host-ip>:8080/sel4.efi.
  • Check same network / subnet.
  • Check UEFI boot order.

Variable test counts (13–127 instead of 141)

Custom BL31 is not properly installed. Verify both A and B slots are flashed. Look for the banner message on boot.
